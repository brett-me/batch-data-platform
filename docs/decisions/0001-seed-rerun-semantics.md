# 0001 - Seed Rerun Semantics

## Status

Accepted

## Context

The local platform currently supports schema application, deterministic seeding, and sanity checks.

At the current stage of the repository, the priority is a simple and predictable local workflow that can be rerun from a known clean state. The synthetic dataset is fully generated by the repository and is intended to be reproducible for the same input parameters.

A decision is required for how repeated seed runs should behave.

## Decision

The platform uses **truncate + reload** semantics for seeded tables.

The seed process will:

- truncate seeded tables
- restart identity counters
- regenerate deterministic data from scratch
- reload the same logical dataset for the same seed and scale inputs

This approach is intentionally chosen over `on conflict` or upsert-based merge logic at the current platform stage.

## Rationale

`truncate + reload` is the simplest and most predictable approach for the current local development workflow.

It is appropriate because:

- the dataset is synthetic and fully controlled by the repository
- deterministic seeding allows the same logical dataset to be recreated reliably
- debugging is easier when reruns begin from a known clean state
- the current schema and workflow do not yet require merge logic or partial rerun handling

This keeps the platform reproducible while avoiding premature complexity.

## Consequences

### Positive

- reruns are easy to reason about
- local rebuilds are fast and predictable
- row counts and seed outputs remain stable for the same inputs
- the current platform remains simple to operate and debug

### Negative

- seeded tables are fully replaced on each rerun
- this is not the final long-term rerun strategy for production-style loads
- merge or upsert semantics will still need to be introduced later where appropriate

## Operational Notes

The current seed process truncates related tables in a single statement and resets identity counters before reloading data.

Where foreign key dependencies exist, truncation must respect table relationships. Child tables must be cleared together with, or before, their parent tables.

## Follow-up

Later phases of the platform should introduce more granular rerun-safe semantics where appropriate, including merge or conflict-handling patterns for datasets that should not be fully replaced on rerun.
